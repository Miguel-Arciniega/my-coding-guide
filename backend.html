Mental model

The request flow

One HTTP request walks down four layers and back. Know which annotation lives at each layer and what it returns — most CRUD questions are just "show me this path for entity X."

LAYER 01@RestController
Controller — maps HTTP to methods. Reads the request (@PathVariable, @RequestParam, @RequestBody), returns a DTO or ResponseEntity. No business logic here.
↓ calls
LAYER 02@Service
Service — business logic + the @Transactional boundary. Orchestrates repositories, maps entity ↔ DTO, throws domain exceptions.
↓ calls
LAYER 03@Repository
Repository — interface … extends JpaRepository<T, ID>. You declare method signatures; Spring Data generates the SQL. No implementation written by you.
↓ persists
LAYER 03@Entity
Entity — a class mapped to a table. Hibernate manages it; inside a transaction, changes flush automatically (dirty checking).

@Query — when names get awkward

JPQL, native, and bulk @Modifying
!
JPQL queries entities (from User u), native queries hit the table (from users). A @Modifying bulk update/delete bypasses the persistence context — stale managed entities won't reflect it unless you set clearAutomatically = true.

Pagination, returned

Pageable in, Page out

Page runs a second COUNT query (knows total pages); Slice skips it (only knows "is there a next page") — cheaper when you don't need totals.

Layer 03 · Mapping

Entity & mapping

A POJO annotated to mirror a table. Get the id strategy, the relationship annotations, and fetch types right — that's 90% of entity questions.

User.java

@GeneratedValue strategies

StrategyHow / when
IDENTITYDB auto-increment column (MySQL, Postgres serial). Most common. Disables JDBC batch inserts — Hibernate must INSERT to learn the id.
SEQUENCEDB sequence (Postgres, Oracle). Allows insert batching; preferred for high write volume.

Relationships & default fetch

AnnotationDefault fetchNotes
@ManyToOneEAGERThe "many" side owns the FK. Set fetch = LAZY explicitly — eager is a silent perf trap.
@OneToOneEAGERAlso eager by default; make it LAZY.
@OneToManyLAZYUse mappedBy on the inverse side. The owning side is the @ManyToOne.
@ManyToManyLAZYJoin table via @JoinTable. Often better modeled as two OneToMany to a link entity.

Cascade & orphan removal

  • cascade = CascadeType.ALL — PERSIST, MERGE, REMOVE, REFRESH, DETACH propagate from parent to children.
  • orphanRemoval = true — deletes a child when it's removed from the parent's collection. Stronger than REMOVE (which only fires on parent delete).
  • @Enumerated(EnumType.STRING) — always use STRING. ORDINAL stores 0,1,2 and silently breaks if you reorder the enum.
→
JPA requires a no-arg constructor (can be protected). Keep it, plus a real constructor for your code.

Layer 02 · Logic

Service & DTO

Where business rules and the transaction boundary live. Map entities to DTOs here so they never leak to the web layer.

Two things to be able to defend

  • Constructor injection over @Autowired fields: fields can be final, the bean is immutable, and you can construct it in a unit test with no Spring context. A single constructor doesn't even need @Autowired.
  • @Transactional semantics: default propagation REQUIRED (joins an existing tx or starts one); readOnly = true on pure reads; rollback fires on unchecked (RuntimeException) by default — checked exceptions need rollbackFor.
⚑
Self-invocation trap: calling one @Transactional method from another method in the same class goes through this, bypassing the Spring proxy — so the annotation does nothing. Call across beans, or it won't apply.

Senior signal

Gotchas that separate seniors

A junior gets the CRUD working; the senior names these before being asked. If you only memorize one tab beyond Repository, make it this one.

1 · N+1 queries

Looping a lazy collection fires 1 query for the parents + N for each child set. The single most-asked JPA performance question.

⚑
Fix: @Query("select u from User u join fetch u.orders"), or @EntityGraph(attributePaths = "orders") on the repo method, or @BatchSize on the collection.

2 · LazyInitializationException

Touching a LAZY field after the persistence context closed — e.g. in the controller, after the service method returned. The session is gone, so Hibernate can't load it.

  • Right fix: fetch it inside the transaction (JOIN FETCH / EntityGraph), or map to a DTO before returning.
  • Wrong "fix": leaning on Open Session In View — name it as the anti-pattern, don't endorse it (see #6).

3 · save() is redundant inside a transaction

A managed entity is flushed by dirty checking on commit. Calling save() on something you just loaded in the same tx works but reveals you don't understand the persistence context. (Cross-reference the Repository tab.)

4 · getReferenceById vs findById

findById hits the DB now and returns the entity. getReferenceById returns a lazy proxy with no SELECT — ideal for setting a foreign key (order.setUser(repo.getReferenceById(id))) when you only need the id.

5 · equals() / hashCode() on entities

Using a DB-generated id in hashCode() breaks entities placed in a Set before they're persisted (id is null, then changes). Use a stable business key, or a Hibernate-safe id-based pattern.

6 · Open Session In View (OSIV)

On by default in Spring Boot. Keeps the Hibernate session open through view/response rendering, which hides N+1 problems and holds DB connections longer. A classic "what would you disable and why?" — answer: spring.jpa.open-in-view=false, then fetch explicitly.

7 · Always return DTOs, never entities

Serializing entities to JSON triggers lazy loads, leaks columns, and causes recursion on bidirectional relationships. Map to a record/DTO (or interface projection) in the service.

8 · jakarta vs javax

Spring Boot 3 moved persistence to jakarta.persistence.*. Importing javax.persistence.* is the fastest way to date yourself in a 2025+ interview.

Clauses & execution order

You write SELECT first, but the engine runs it almost last. Knowing the real order explains every "why can't I use that alias here" question.

Consequence: SELECT runs after WHERE/GROUP BY, so a column alias defined in SELECT can't be used in WHERE or GROUP BY — but it can in ORDER BY (which runs last). That alias question is a frequent gotcha.

WHERE vs HAVING

ClauseFiltersAggregates allowed?
WHEREIndividual rows, before groupingNo — runs before GROUP BY
HAVINGGroups, after GROUP BYYes — HAVING COUNT(*) > 1

Rule of thumb: filter on a raw column → WHERE. Filter on an aggregate → HAVING. Use both together freely.

Combining tables

Joins

A join matches rows across tables on a condition. The type decides what happens to rows with no match.

JoinKeeps
INNER JOINOnly rows matching on both sides.
LEFT JOINAll left rows; right columns NULL when no match. (Most common after INNER.)
RIGHT JOINAll right rows; mirror of LEFT. Rarely needed — flip the tables and use LEFT.
FULL OUTER JOINAll rows from both; NULLs where either side is missing. (Not in MySQL — emulate with UNION.)
CROSS JOINCartesian product — every left row × every right row. No ON clause.
SELF JOINA table joined to itself with aliases. Hierarchies: employee ↔ manager.

Inner vs left, side by side

same query, two join types
→
Anti-join idiom: LEFT JOIN ... WHERE right.id IS NULL returns left rows with no right match. Putting a right-table filter in WHERE (not ON) silently turns a LEFT JOIN back into an INNER JOIN — a common bug.

Aggregates & grouping

Aggregate functions collapse many rows into one value; GROUP BY does it per group.

FunctionNote

COUNT(col)Non-NULL values of col only.

SUM / AVGSkip NULLs. AVG divides by non-NULL count, not row count.
MIN / MAXWork on numbers, dates, and strings.

EXISTS vs IN — and the NOT IN null trap

NOT EXISTS is null-safe; NOT IN is not
!
NOT IN with a NULL in the list returns no rows. If the subquery yields any NULL, x NOT IN (1, 2, NULL) evaluates to unknown for every x. Prefer NOT EXISTS for anti-joins — it's null-safe and usually as fast or faster.

CTEs — readability over nesting

WITH ... AS (...)

A CTE doesn't change results vs a derived table — it makes multi-step logic legible and lets you reference the same intermediate set more than once.

Keys & constraints

  • PRIMARY KEY — unique + NOT NULL, one per table. Often the clustered index.
  • UNIQUE — enforces uniqueness, allows (typically one) NULL, multiple per table.
  • FOREIGN KEY — references another table's PK/unique column; enforces referential integrity.

Indexes

  • Speed up reads/filters/joins; slow down writes and cost storage — index selectively.
  • Clustered — defines the physical row order; one per table (the PK by default in many engines).
  • Non-clustered — separate structure pointing to rows; many per table.
  • Skip indexing low-cardinality columns (e.g. a boolean) and tiny or write-heavy tables.

UNION vs UNION ALL · Normal forms · NULLs

  • UNION removes duplicates (extra sort/dedup cost); UNION ALL keeps everything and is faster. Both require matching column counts/types.
  • 1NF atomic values, no repeating groups · 2NF 1NF + no partial dependency on part of a composite key · 3NF 2NF + no transitive dependency (non-key depending on non-key). Denormalize deliberately for read speed.
  • NULL is unknown: NULL = NULL is not true — use IS NULL. Default it with COALESCE(x, 0).

Coding

Two complementary tools: Pattern Recognition trains the reflex of mapping an unseen problem to the right approach; the Visualizer steps the optimal solution through real Java, frame by frame.

How to use this

Read this tab once before anything else. The diagnosis protocol and the two lookup tables below are the actual product — the patterns in Phase 1 are downstream of them.

What TCS's coding round actually tests. For experienced Java hires it is typically one problem in a fixed window (often ~20–30 min), with limited compile attempts, where you write the whole program and print exact output. The bar is correct, working, edge-case-clean code on an easy/medium problem — not exotic algorithms. Optimize for: reading the constraints, picking the pattern fast, and not wasting compiles on I/O or formatting bugs.

The diagnosis protocol — run this on every problem

  1. Characterize the input

    Array / string / tree / graph / intervals / linked list? Is it sorted? What is the size bound n? The size alone narrows the pattern before you read the prose (see the complexity table below).

  2. Characterize the ask

    Exist / find one? Count how many? Optimize (max / min / longest / shortest)? Enumerate all arrangements? Shortest path? Each verb maps to a small family of patterns.

  3. Write brute force, name its bottleneck

    State the obvious O(n²) or exponential solution in one line, then name what makes it slow. The pattern almost always exists to kill that specific bottleneck: nested loop searching for a complement → hashing; recomputing a range sum → prefix sum; linear scan on sorted data → binary search; recomputing the same subproblem → memoize.

  4. Map signal → pattern, then check the precondition

    Use the cheat sheet below to get a candidate. Then verify its precondition holds — two pointers needs sortedness or a monotonic property; binary-search-on-answer needs the feasibility function to be monotonic. If the precondition fails, the pattern is wrong.

Java I/O boilerplate — memorize this cold

The TCS compiler round penalizes formatting/I/O mistakes and limits your compiles. Scanner is fine for tiny inputs but slow; BufferedReader is the safe default. Print with a StringBuilder to avoid per-line flush cost and stray characters.

import java.util.*;
import java.io.*;

public class Main {
    public static void main(String[] args) throws IOException {
        BufferedReader br = new BufferedReader(new InputStreamReader(System.in));
        int n = Integer.parseInt(br.readLine().trim());        // first line: count
        StringTokenizer st = new StringTokenizer(br.readLine()); // second line: the values
        int[] a = new int[n];
        for (int i = 0; i < n; i++) a[i] = Integer.parseInt(st.nextToken());

        StringBuilder sb = new StringBuilder();
        // ... compute, then: sb.append(result).append('\n');
        System.out.print(sb);                                  // print() not println(): no extra trailing newline
    }
}

Phase 1 — Theory & templates

Do not solve these. Read each card, then type the template out once from a blank file so your hands know it. Each card gives the recognition triggers (what to look for), the template, one worked example, complexity, and the pitfall that costs people the problem. Tier badge = priority.

1 Hashing — frequency & lookupTier 1▸
Recognize when
  • You need "has this value/complement appeared before?" → trade memory for time, drop a nested loop to O(n).
  • Counting occurrences, deduping, grouping by a key, anagram checks (count letters).
Worked example — Two Sum (unsorted, return indices)
int[] twoSum(int[] nums, int target) {
    Map<Integer,Integer> seen = new HashMap<>();      // value -> index
    for (int i = 0; i < nums.length; i++) {
        int need = target - nums[i];
        if (seen.containsKey(need)) return new int[]{seen.get(need), i};
        seen.put(nums[i], i);
    }
    return new int[]{-1,-1};
}
Why it works

The brute force checks every pair (O(n²)). Instead of re-searching for the complement each time, you remember every value you've seen in a map; the lookup is O(1), so the whole scan is O(n).

Complexity
Time O(n)Space O(n)
Pitfall: put the current value into the map after checking for its complement, or you'll match an element with itself.
2 Two PointersTier 1▸
Recognize when
  • Input is sorted (or you can sort it) and you want a pair/triplet meeting a target.
  • Comparing from both ends (palindrome, container area), or partitioning in place.
  • Two sequences merged in order; or a slow/fast pair to drop duplicates.
Worked example — pair sum in a sorted array
int[] pairSum(int[] a, int target) {   // a is sorted ascending
    int lo = 0, hi = a.length - 1;
    while (lo < hi) {
        int sum = a[lo] + a[hi];
        if (sum == target) return new int[]{lo, hi};
        if (sum < target) lo++;            // need bigger -> move left bound up
        else hi--;                          // need smaller -> move right bound down
    }
    return new int[]{-1,-1};
}
Why it works

Sortedness gives a monotonic relationship: if the sum is too small, the only way to grow it is to raise the low pointer; too big, lower the high pointer. Each step eliminates one element, so it's a single O(n) pass instead of O(n²).

Complexity
Time O(n) (plus O(n log n) if you sort)Space O(1)
Pitfall: the monotonic argument only holds on sorted data. On unsorted input, use hashing instead.
3 Sliding WindowTier 1▸
Recognize when
  • The answer is a contiguous subarray/substring, and you want longest/shortest/max/min/count under a constraint.
  • Fixed window: "of size k". Variable window: "longest/shortest such that <condition>".
Worked example — longest substring without repeating characters (variable window)
int longestUnique(String s) {
    Map<Character,Integer> last = new HashMap<>();  // char -> most recent index
    int start = 0, best = 0;
    for (int end = 0; end < s.length(); end++) {
        char c = s.charAt(end);
        if (last.containsKey(c) && last.get(c) >= start)
            start = last.get(c) + 1;          // shrink from the left past the duplicate
        last.put(c, end);
        best = Math.max(best, end - start + 1);  // window length
    }
    return best;
}
Why it works

Both pointers only move forward, so each element enters and leaves the window once → O(n). The window holds the current valid candidate; you grow it on the right and shrink it on the left only when the constraint breaks.

Complexity
Time O(n)Space O(k) (alphabet / window contents)
Pitfall: when shrinking, only jump start forward — never backward — or you'll re-count characters and the window logic collapses.
4 Prefix SumTier 1▸
Recognize when
  • Repeated range-sum queries, or "subarray summing to K / divisible by K / with property P".
  • Any time you'd otherwise recompute overlapping sums in a loop.
Worked example — count subarrays summing to K (prefix + hashing)
int subarraySumK(int[] nums, int k) {
    Map<Integer,Integer> count = new HashMap<>(); // prefixSum -> how many times seen
    count.put(0, 1);                              // empty prefix
    int sum = 0, res = 0;
    for (int n : nums) {
        sum += n;
        res += count.getOrDefault(sum - k, 0);    // a prior prefix that makes the gap = k
        count.merge(sum, 1, Integer::sum);
    }
    return res;
}
Why it works

Sum of nums[i..j] = prefix[j] − prefix[i−1]. Asking "which subarrays sum to k" becomes "for each prefix, how many earlier prefixes equal sum − k" — an O(1) map lookup per element. This combines two Tier-1 patterns, which is common.

Complexity
Time O(n)Space O(n)
Pitfall: forgetting the count.put(0,1) seed — you'll miss subarrays that start at index 0.
5 Binary Search (array & on answer)Tier 1▸
Recognize when
  • Sorted input + find/insert position, or O(log n) is explicitly required.
  • On the answer: "minimize the maximum / maximize the minimum / smallest capacity/speed/days such that it works." The feasibility of a candidate answer is monotonic.
Template A — classic search on a sorted array
int search(int[] a, int target) {
    int lo = 0, hi = a.length - 1;
    while (lo <= hi) {
        int mid = lo + (hi - lo) / 2;   // avoids overflow vs (lo+hi)/2
        if (a[mid] == target) return mid;
        if (a[mid] < target) lo = mid + 1;
        else hi = mid - 1;
    }
    return -1;
}
Template B — binary search on the answer space
// smallest x in [lo, hi] for which feasible(x) is true; feasible must be monotonic
int minFeasible(int lo, int hi) {
    while (lo < hi) {
        int mid = lo + (hi - lo) / 2;
        if (feasible(mid)) hi = mid;        // mid works -> answer is mid or smaller
        else lo = mid + 1;                 // mid fails -> need bigger
    }
    return lo;
}
Why it works

Each step halves the search space. The leap most people miss: the "space" doesn't have to be an array index — it can be the range of possible answers (e.g. eating speeds from 1 to max pile). If "answer x works ⇒ every x′ > x also works," the boolean is sorted and binary search applies.

Complexity
Time O(log n) · on-answer O(n log(range))Space O(1)
Pitfall: off-by-one infinite loops. Pick one template and keep its invariant straight: lo <= hi with mid±1 for exact-match; lo < hi with hi = mid for boundary-finding.
6 Sort + GreedyTier 1▸
Recognize when
  • Sorting first makes a locally-optimal choice obvious and safe.
  • "Maximum number of non-overlapping", "minimum to remove", "earliest deadline", "assign smallest that fits".
Worked example — max non-overlapping intervals (activity selection)
int maxNonOverlapping(int[][] intervals) {
    Arrays.sort(intervals, (x, y) -> x[1] - y[1]);   // by END time, ascending
    int count = 0, lastEnd = Integer.MIN_VALUE;
    for (int[] iv : intervals) {
        if (iv[0] >= lastEnd) {          // starts after the last one we kept ends
            count++;
            lastEnd = iv[1];
        }
    }
    return count;
}
Why it works

Picking the interval that ends earliest leaves the most room for the rest — a provably optimal greedy choice. The danger with greedy is that the local choice isn't always safe; the sort key is what makes it safe here.

Complexity
Time O(n log n)Space O(1)
Pitfall: sorting by start time instead of end time. It feels natural and is wrong — it can pick a long early interval that blocks several short ones.
7 Stack (matching & monotonic)Tier 1▸
Recognize when
  • Brackets / nesting / "valid sequence"; evaluating expressions; an undo-style most-recent-first need.
  • Monotonic stack: "next/previous greater or smaller element", "how far until a taller bar".
Worked example A — valid parentheses
boolean isValid(String s) {
    Deque<Character> st = new ArrayDeque<>();
    Map<Character,Character> pair = Map.of(')','(', ']','[', '}','{');
    for (char c : s.toCharArray()) {
        if (pair.containsKey(c)) {           // a closing bracket
            if (st.isEmpty() || st.pop() != pair.get(c)) return false;
        } else st.push(c);                  // an opening bracket
    }
    return st.isEmpty();
}
Worked example B — next greater element (monotonic stack)
int[] nextGreater(int[] a) {
    int[] res = new int[a.length];
    Arrays.fill(res, -1);
    Deque<Integer> st = new ArrayDeque<>();   // holds INDICES, values decreasing
    for (int i = 0; i < a.length; i++) {
        while (!st.isEmpty() && a[st.peek()] < a[i])
            res[st.pop()] = a[i];            // a[i] is the answer for everything smaller below it
        st.push(i);
    }
    return res;
}
Why it works

A stack captures "most recent unresolved thing". For brackets, the top is the only opener a closer can legally match. For the monotonic version, each index is pushed and popped once → O(n), and the stack invariant guarantees the first taller element you meet is the true "next greater".

Complexity
Time O(n)Space O(n)
Pitfall: use ArrayDeque, not the legacy java.util.Stack (synchronized, slower). And in Java, st.pop() != pair.get(c) compares Character objects — fine for the small cached range, but prefer !Objects.equals(...) or unbox explicitly if values can exceed 127.
8 Linked List Two-PointerTier 2▸
Recognize when
  • Reverse / reorder a list; detect a cycle; find the middle; find the kth node from the end.
  • A dummy head node simplifies edge cases when the head itself may change.
Worked example A — reverse a linked list
ListNode reverse(ListNode head) {
    ListNode prev = null, cur = head;
    while (cur != null) {
        ListNode next = cur.next;   // save before we overwrite
        cur.next = prev;            // flip the pointer
        prev = cur;                 // advance both
        cur = next;
    }
    return prev;                    // new head
}
Worked example B — cycle detection (Floyd's tortoise & hare)
boolean hasCycle(ListNode head) {
    ListNode slow = head, fast = head;
    while (fast != null && fast.next != null) {
        slow = slow.next;           // +1
        fast = fast.next.next;      // +2
        if (slow == fast) return true;  // they meet iff there's a cycle
    }
    return false;
}
Why it works

Reversal is pure pointer bookkeeping — the only trick is caching next before you clobber it. The fast/slow trick works because a 2× speed pointer laps a 1× pointer inside any loop; the same idea finds the middle (fast hits the end when slow is halfway).

Complexity
Time O(n)Space O(1)
Pitfall: dereferencing fast.next.next without the fast != null && fast.next != null guard → NullPointerException on even-length lists.
9 Tree Traversal (DFS & BFS)Tier 2▸
Recognize when
  • "Level by level / width / right-side view / zigzag" → BFS with a queue.
  • "Depth / path sum / validate / lowest common ancestor / mirror" → DFS recursion.
Worked example A — level-order traversal (BFS)
List<List<Integer>> levelOrder(TreeNode root) {
    List<List<Integer>> res = new ArrayList<>();
    if (root == null) return res;
    Queue<TreeNode> q = new LinkedList<>();
    q.add(root);
    while (!q.isEmpty()) {
        int size = q.size();             // freeze this level's count
        List<Integer> level = new ArrayList<>();
        for (int i = 0; i < size; i++) {
            TreeNode node = q.poll();
            level.add(node.val);
            if (node.left != null) q.add(node.left);
            if (node.right != null) q.add(node.right);
        }
        res.add(level);
    }
    return res;
}
Worked example B — max depth (DFS recursion)
int maxDepth(TreeNode root) {
    if (root == null) return 0;                 // base case
    return 1 + Math.max(maxDepth(root.left), maxDepth(root.right));
}
Why it works

The "freeze q.size() at the top of the loop" trick is what separates levels in BFS. DFS on trees is just recursion with a base case at null; most tree problems reduce to "combine the answers from the left and right subtrees".

Complexity
Time O(n)Space O(width) BFS · O(height) DFS
Pitfall: deep/skewed trees can blow the recursion stack in DFS; if depth can approach n, convert to an explicit stack.
10 Recursion & BacktrackingTier 2▸
Recognize when
  • "Return / count all combinations / permutations / subsets / valid arrangements / paths."
  • Constraint satisfaction (Sudoku, word search, N-Queens). Small n (the size table says ≤ ~20).
Worked example — all subsets
List<List<Integer>> subsets(int[] nums) {
    List<List<Integer>> res = new ArrayList<>();
    backtrack(nums, 0, new ArrayList<>(), res);
    return res;
}
void backtrack(int[] nums, int start, List<Integer> cur, List<List<Integer>> res) {
    res.add(new ArrayList<>(cur));            // every prefix is a valid subset; copy it
    for (int i = start; i < nums.length; i++) {
        cur.add(nums[i]);                     // choose
        backtrack(nums, i + 1, cur, res);     // explore
        cur.remove(cur.size() - 1);          // un-choose (the backtrack)
    }
}
Why it works

The shape is always choose → explore → un-choose. The start index prevents revisiting earlier elements (so you get combinations, not duplicates). For permutations you'd track a used[] boolean array instead of start.

Complexity
Subsets O(2ⁿ · n) · permutations O(n! · n)Space O(n) recursion
Pitfall: adding cur directly to res instead of a copy — every entry then points at the same list that keeps mutating. Always new ArrayList<>(cur).
11 Intervals (sort + sweep)Tier 2▸
Recognize when
  • Ranges that may overlap; merge them, insert one, or count maximum simultaneous overlap.
  • "Minimum meeting rooms", "can attend all meetings", "free time".
Worked example — merge overlapping intervals
int[][] merge(int[][] intervals) {
    Arrays.sort(intervals, (x, y) -> x[0] - y[0]);    // by START
    List<int[]> out = new ArrayList<>();
    for (int[] iv : intervals) {
        if (out.isEmpty() || out.get(out.size()-1)[1] < iv[0])
            out.add(iv);                                  // no overlap -> new block
        else
            out.get(out.size()-1)[1] =
                Math.max(out.get(out.size()-1)[1], iv[1]); // overlap -> extend the end
    }
    return out.toArray(new int[0][]);
}
Why it works

After sorting by start, any overlap must be with the immediately preceding kept interval, so one linear pass suffices. For "max concurrent" problems, a different sweep — sort start and end events separately and walk them — counts overlaps without building the merged set.

Complexity
Time O(n log n)Space O(n)
Pitfall: deciding whether touching endpoints ([1,2] and [2,3]) count as overlapping — read the problem; it changes < to <=.
12 Heap / Priority QueueTier 2▸
Recognize when
  • "Top / smallest / K closest"; "merge K sorted lists"; streaming median; repeatedly extract the min/max.
  • You need partial ordering (the best k) but not a full sort.
Worked example — kth largest element (min-heap of size k)
int kthLargest(int[] nums, int k) {
    PriorityQueue<Integer> heap = new PriorityQueue<>(); // min-heap by default
    for (int n : nums) {
        heap.offer(n);
        if (heap.size() > k) heap.poll();   // keep only the k largest; smallest falls out
    }
    return heap.peek();                     // the smallest of the top-k = kth largest
}
Why it works

A size-k min-heap always holds the k biggest seen so far; the root is the kth largest. Sorting is O(n log n) and wasteful when k is small — this is O(n log k). For max-behavior use new PriorityQueue<>(Collections.reverseOrder()) or a comparator.

Complexity
Time O(n log k)Space O(k)
Pitfall: mixing up the heap direction. To keep the k largest, you evict with a min-heap. The intuition feels backwards the first few times — write the invariant down.
13 Dynamic ProgrammingTier 3▸
Recognize when
  • "Count the number of ways", "min/max cost or length", "can you reach / partition / make change".
  • A brute-force recursion exists but recomputes the same (index, state) repeatedly (overlapping subproblems + optimal substructure).
Worked example — house robber (1D DP, space-optimized)
int rob(int[] nums) {
    int prev2 = 0, prev1 = 0;            // best up to i-2 and i-1
    for (int n : nums) {
        int cur = Math.max(prev1, prev2 + n); // skip this house vs rob it
        prev2 = prev1;
        prev1 = cur;
    }
    return prev1;
}
Recognition workflow (more useful than memorizing recurrences)

1) Write the recursive brute force and identify the state (what arguments change). 2) If the same state recurs, add a memo (Integer[] dp, fill with null). 3) Optionally flip to bottom-up tabulation. 4) Optionally drop the table to a few rolling variables, as above. For 2D problems (grids, two strings — edit distance, LCS) the state is a pair (i, j) and the table is a 2D array.

Complexity
1D O(n) · 2D O(n·m)Space O(1)→O(n·m)
Pitfall (and why this is Tier 3 for TCS): DP recognition is the hardest to fake under time pressure. Target "I can identify the state and write the memoized recursion for classic 1D problems" — don't burn days on hard 2D DP you're unlikely to be gated on.
14 Graph BFS / DFSTier 3▸
Recognize when
  • A grid or explicit graph; "number of connected regions / components / islands".
  • "Shortest number of steps" in an unweighted graph → BFS. "Ordering with prerequisites" → topological sort.
Worked example — number of islands (grid DFS / flood fill)
int numIslands(char[][] grid) {
    int count = 0;
    for (int r = 0; r < grid.length; r++)
        for (int c = 0; c < grid[0].length; c++)
            if (grid[r][c] == '1') { sink(grid, r, c); count++; }
    return count;
}
void sink(char[][] g, int r, int c) {
    if (r < 0 || c < 0 || r >= g.length || c >= g[0].length || g[r][c] != '1') return;
    g[r][c] = '0';                       // mark visited in place
    sink(g, r+1, c); sink(g, r-1, c);
    sink(g, r, c+1); sink(g, r, c-1);  // 4-directional flood
}
Why it works

Each new '1' you encounter starts a fresh region; the flood fill consumes the entire connected piece so you never double-count. For shortest-path-in-steps, swap DFS for a BFS queue and count levels — BFS reaches nodes in increasing distance order, which DFS does not.

Complexity
Time O(rows · cols) / O(V + E)Space O(rows · cols)
Pitfall: forgetting to mark nodes visited (here: overwriting to '0') → infinite recursion. On a graph with an adjacency list, keep a separate visited set.

Awareness-only (don't drill for TCS, but recognize the name): Union-Find / DSU (connectivity, merging groups, cycle detection in undirected graphs), Trie (prefix lookups, autocomplete), Bit manipulation (XOR for the "single number", masks for subsets), Dijkstra (shortest path with weights — a heap-driven BFS). If you see one of these in the round, you'll likely have time to look up the template; you just need to recognize which one it is.

Phase 3 — Solo (pattern hidden)

No pattern is given. Run the diagnosis protocol from the Start Here tab, commit to an approach, then code it. Only after you've attempted it should you open "What pattern was this?" to check your diagnosis. Mediums first, hards at the end — those are your "couple of hards".

Use a timer: 25 min for mediums, 40 for hards. If you're stuck at the time limit, open the reveal, see the pattern (not the solution), and re-attempt. The miss-then-see-the-pattern loop is exactly what builds recognition — a clean solve teaches less than a corrected wrong guess.

Mediums — diagnose then solve

Product of Array Except SelfMediumLeetCode 238
What pattern was this?
Prefix / suffix products. The "no division, O(n)" constraint is the tell. Build running products from the left, then multiply by running products from the right in a second pass.
Find All Anagrams in a StringMediumLeetCode 438
What pattern was this?
Sliding window + hashing. "All substrings of fixed length that are anagrams" → fixed-size window over a 26-count signature; slide and compare counts.
Search in Rotated Sorted ArrayMediumLeetCode 33
What pattern was this?
Binary search (modified). Sorted-ish + O(log n) demanded. At each step one half is properly sorted — figure out which, and whether the target lies in it.
Word SearchMediumLeetCode 79
What pattern was this?
Backtracking on a grid (DFS). "Does a path exist spelling the word" → DFS from each cell, mark visited, recurse on neighbors, un-mark on the way out.
Rotting OrangesMediumLeetCode 994
What pattern was this?
Multi-source BFS. "Minimum minutes for spread" = shortest time in an unweighted grid → BFS from all rotten cells at once, counting levels.
Longest Palindromic SubstringMediumLeetCode 5
What pattern was this?
Expand around center (or 2D DP). For each of the 2n−1 centers, expand outward while characters match. The "contiguous substring with a property" framing hints at a window-like scan; here the center-expansion variant is cleanest.
Longest Increasing SubsequenceMediumLeetCode 300
What pattern was this?
DP (O(n²)) — dp[i] = LIS ending at i. The O(n log n) version uses binary search over a "tails" array; recognizing the binary-search upgrade is a strong signal.

Hards — your stretch set

Trapping Rain WaterHardLeetCode 42
What pattern was this?
Two pointers (or monotonic stack, or prefix max arrays). Water above each bar = min(maxLeft, maxRight) − height. Two pointers converging from both ends does it in O(1) space.
Minimum Window SubstringHardLeetCode 76
What pattern was this?
Variable sliding window + hashing. "Shortest substring containing all of T" → grow the right to become valid, then shrink the left as far as still-valid, recording the minimum. The hardest version of card 3.
Word LadderHardLeetCode 127
What pattern was this?
BFS shortest path. "Fewest transformations" where each one-letter change is an edge → BFS over the implicit word graph; the first time you reach endWord is the shortest ladder.
Merge k Sorted ListsHardLeetCode 23
What pattern was this?
Heap (k-way merge). "Merge K sorted" is the canonical heap signal — a size-k min-heap of the current front of each list; poll the smallest, push its successor.
Edit DistanceHardLeetCode 72
What pattern was this?
2D DP. Two strings + "minimum operations" → state (i, j) over both prefixes; each cell is the best of insert/delete/replace. The archetypal 2D table. If you only learn one hard DP, learn this shape.

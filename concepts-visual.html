The five SOLID principles

Five pillars holding up changeable OO code. Read the column letters top-to-bottom: S-O-L-I-D.

S🎯Single ResponsibilityOne class, one reason to change.
O🔌Open / ClosedExtend by adding, not editing.
L🔄Liskov SubstitutionSubtypes swap in safely.
I✂️Interface SegregationSmall interfaces, no dead methods.
D🔻Dependency InversionDepend on abstractions.
↓   all five serve one goal:   loose coupling — change one thing without breaking five

Hook: the word SOLID is the mnemonic. Each pillar removes a different reason that a change would ripple outward.

Open/Closed — modify vs extend

The same feature, two ways. On the left you reopen tested code for every new case; on the right you plug in a new class and never touch the old code.

✗ closed mindset — modify

One method, edited forever

A growing switch on shape type. Each new shape reopens this method and risks the cases that already passed.

switch(shape) case Circle case Square + case Triangle ✎ edit!
✓ open for extension

One interface, many plug-ins

Each shape owns its area(). A new shape is a new file; the calculator stays closed.

interface Shape
↳ implements Circle Square Triangle ✨ just added

Picture a power strip: you extend it by plugging in new devices — you never rewire the strip itself.

How microservices communicate

Solid arrows = synchronous (request → wait → response, over REST/gRPC). Dashed arrows = asynchronous (publish an event to a broker; subscribers react on their own time).

synchronous — REST / gRPC (caller waits) asynchronous — event via broker (fire & forget)

Phone call vs text message: sync = both on the line now; async = send it and move on. Wrap both with a gateway, service discovery, and a circuit breaker.

4

A HashMap collision

Two distinct keys compute the same bucket index. Java keeps both: it chains them in a list, and once a bucket grows past 8 it upgrades that bucket to a red-black tree.

Hook: "8 turns a list into a tree." Collisions are normal; a good hashCode() + the 0.75 resize keep them rare, so lookups stay ~O(1).

5

SQL — written order vs execution order

You write SELECT first, but the engine runs it almost last. This single mismatch explains every "why can't I use that alias / aggregate here" gotcha.

How you write it
SELECT FROM WHERE GROUP BY HAVING ORDER BY
How the engine runs it
1FROM / JOIN 2WHERE 3GROUP BY 4HAVING 5SELECT 6ORDER BY

Consequences: WHERE runs before grouping → it can't see aggregates (use HAVING). SELECT runs at step 5 → a column alias isn't visible to WHERE/GROUP BY, but is visible to ORDER BY (step 6).

Mnemonic: "From Where Groups Have Selected Order." The query on tab 5 of the study page rides exactly this pipeline.

7

Inversion of Control

Who creates the dependency? Without IoC, the class does (new). With IoC, the container does, and injects it — so control is inverted from your code to the framework.

✗ without IoC — tight coupling

The class wires itself

OrderService
contains →new StripeGateway()

Hard-wired to one implementation. Can't swap it, can't inject a mock to test.

✓ with IoC — loose coupling

The container wires it

Spring IoC container
StripeGatewayPayPalGatewayOrderService
⬇ injects PaymentGateway
OrderService(PaymentGateway)

Depends on the PaymentGateway interface; the container chooses the bean. Swap or mock freely.

The Hollywood Principle: "Don't call us, we'll call you." DI is the mechanism; IoC is the inversion. This is SOLID's "D" delivered as a framework feature.

Object-oriented design

What are the SOLID principles?

Five class-design principles (acronym by Michael Feathers, principles popularised by Robert C. "Uncle Bob" Martin) that keep object-oriented code loosely coupled, easy to extend, and safe to change. They're guidelines, not laws — the goal each one serves is "change one thing without breaking five others."

SSingle ResponsibilityA class has one reason to change — one job, one actor it answers to.
OOpen / ClosedOpen for extension, closed for modification. Add behaviour with new code.
LLiskov SubstitutionA subtype must be usable anywhere its base type is, without surprises.
IInterface SegregationMany small, focused interfaces beat one fat one. No unused methods.
DDependency InversionDepend on abstractions, not concretions. High-level owns the interface.

One line each — what they actually mean

LetterPrincipleSmell it fixes
SSingle Responsibility — split a class that mixes, say, persistence + formatting + business rules."God class" that changes for many unrelated reasons.
OOpen/Closed — extend via new classes / strategies instead of editing tested code.A growing switch / if-else edited for every new case.
LLiskov Substitution — subclasses honour the base contract (no weaker outputs, no stronger inputs, no thrown "unsupported").Square extends Rectangle breaking setWidth; throw UnsupportedOperationException in an override.
IInterface Segregation — clients depend only on methods they use.Implementing a huge interface with half the methods left empty.
DDependency Inversion — both layers depend on an abstraction; the container wires the concrete one in.new ConcreteThing() hard-wired inside business logic.

How they connect (this is the senior answer)

They aren't five unrelated rules — they reinforce each other. Dependency Inversion tells you to depend on an interface; Open/Closed is what that buys you (swap implementations without editing callers); Liskov guarantees the swap is safe; Interface Segregation keeps those abstractions small enough to honour; Single Responsibility is the reason you had separate classes to wire together in the first place. In Spring, the framework's IoC container is literally Dependency Inversion as a product feature.

🎤 Say thisSOLID is five principles for keeping OO code changeable: Single Responsibility — one reason to change; Open/Closed — extend without modifying; Liskov — subtypes are safely substitutable; Interface Segregation — small focused interfaces; Dependency Inversion — depend on abstractions. The thread through all of them is loose coupling, so a change stays local.
Which one does Spring help you follow automatically?
Dependency Inversion (and Open/Closed). Spring's IoC container injects an abstraction (PaymentGateway) and decides the concrete bean, so your high-level code never says new StripeGateway().
Give the classic Liskov violation.
Square extends Rectangle: calling setWidth(5) on a square silently changes its height too, so code written for a Rectangle breaks. The fix is to not model it as inheritance. Another tell-tale: an override that throws UnsupportedOperationException.
Isn't SRP just "small classes"?
No — it's "one reason to change," i.e. one stakeholder/axis of change. A class can be large and still have a single responsibility, or tiny and still serve two masters (e.g. mixing report formatting with the calculation rules).

SOLID, up close — the "O"

The Open/Closed Principle & how it's used

Bertrand Meyer's rule: software entities should be open for extension, but closed for modification. You should be able to add new behaviour by adding new code, not by reopening and editing code that already works and is already tested. The mechanism that makes it possible is abstraction + polymorphism.

The smell: a switch you keep reopening

Every new shape forces you back into the same method — re-testing and risking the cases that already worked.

AreaCalculator.java — violates OCP

The fix: push behaviour behind an abstraction

Let each type own its own implementation. Adding Triangle is a brand-new file; the calculator and every caller stay untouched (closed).

Shape.java + implementations — open for extension
→
How it shows up day to day: the Strategy pattern, plug-in architectures, Spring injecting a List<Validator> you loop over, payment providers behind a PaymentGateway interface, a @Bean you add without touching existing config. All "add a class, don't edit one."

When NOT to apply it

  • Don't pre-abstract on speculation. Adding interfaces "in case it changes" is over-engineering. Apply OCP at the axis where change has actually proven likely (the "shape type" really does keep growing).
  • Rule of thumb: write it concrete the first time, abstract on the second variation, mandatory by the third.
🎤 Say thisOpen for extension, closed for modification. I add new behaviour by adding a new class behind an interface — like a new Shape implementation — instead of editing a switch in tested code. Polymorphism is what makes the existing code closed while the system stays extensible.
Which design patterns embody OCP?
Strategy, Template Method, Decorator, Observer, and Factory all let you add variants without editing existing code. Strategy is the cleanest one-liner example.
How does Spring make OCP easy?
Inject the abstraction and let new implementations register themselves. e.g. @Autowired List<Notifier> picks up every Notifier bean — drop in SmsNotifier as a new @Component and the dispatching code never changes.
Can you ever truly never modify code?
No — it's a direction, not an absolute. Bug fixes modify code. OCP targets foreseeable extension points (new payment type, new export format), not every line.

Distributed systems

How do microservices communicate?

Two families: synchronous (caller waits for a response) and asynchronous (caller fires a message and moves on). Picking the right one per interaction — and naming the supporting infrastructure — is the whole answer.

The two styles

StyleHowUse when
SynchronousREST/HTTP (JSON, ubiquitous) · gRPC (HTTP/2 + Protobuf, fast, typed, streaming) · GraphQL (client picks fields).You need the answer now to continue — a read, a query, a request/response.
AsynchronousMessaging / events via a broker — Kafka (log/stream), RabbitMQ/ActiveMQ (queues), SQS/SNS. Pub/sub or point-to-point queues.Fire-and-forget, fan-out, decoupling, buffering spikes, event-driven workflows.

Synchronous — a declarative REST call

Spring Cloud OpenFeign — call another service like a local method

Spring options: RestTemplate (legacy, blocking), WebClient (reactive, the modern default), and @FeignClient (declarative — least boilerplate).

Asynchronous — publish an event, don't wait

Kafka — producer fires, a different service reacts on its own time

Sync vs async — the trade-off they're really asking about

SynchronousAsynchronous
CouplingTight — caller needs callee up nowLoose — broker decouples them in time
FailureCascades; one slow service stalls the chainIsolated; messages buffer until consumed
ConsistencyImmediate / strongEventual
ComplexitySimple to reason about & debugHarder — ordering, idempotency, dead letters

The supporting cast (drop these names)

  • API Gateway — single entry point; routing, auth, rate-limiting (Spring Cloud Gateway).
  • Service Discovery — services find each other by name, not hard-coded host (Eureka, Consul).
  • Load balancing — spread calls across instances (client-side: Spring Cloud LoadBalancer).
  • Circuit Breaker — stop hammering a failing service; fail fast + fallback (Resilience4j).
  • Saga — distributed transaction as a chain of local transactions + compensations (no 2-phase commit).
  • Service Mesh — push retries/mTLS/observability into a sidecar (Istio, Linkerd).
🎤 Say thisTwo ways: synchronous request/response over REST or gRPC when I need the answer immediately, and asynchronous messaging over a broker like Kafka or RabbitMQ when I want to decouple and fan out. I default to async for cross-service workflows because it isolates failures and absorbs load; I keep sync for reads. Around both I'd put an API gateway, service discovery, and a circuit breaker.
REST vs gRPC — when gRPC?
gRPC for high-throughput internal service-to-service calls: binary Protobuf is smaller/faster, it's strongly typed via a shared .proto, and it supports streaming over HTTP/2. REST for public/browser-facing APIs and human debuggability.
How do you handle a transaction across services?
You don't use one DB transaction. Use the Saga pattern: a sequence of local transactions, each publishing an event that triggers the next; on failure you run compensating transactions to undo. Orchestration (a coordinator) or choreography (services react to events).
What stops a failure in one service taking down the rest?
A circuit breaker (Resilience4j): after N failures it "opens" and fails fast with a fallback instead of blocking threads, then half-opens to test recovery. Plus timeouts, bulkheads, and retries with backoff.
Async means a message might arrive twice — now what?
Make consumers idempotent: dedupe on a message/event id, or design the operation so re-applying it is a no-op. Pair with a dead-letter queue for messages that keep failing.

Data structures · Java internals

What is a HashMap collision?

A collision is when two distinct keys map to the same bucket in the underlying array — either because they share a hashCode(), or because two different hashes reduce to the same index. Collisions are normal and unavoidable; what matters is how the map resolves them.

Why it happens

A HashMap is an array of buckets. To place a key it computes index = (n - 1) & hash(key), where n is the table size. Since there are far more possible keys than buckets, different keys can land on the same index — a collision.

a real Java String collision — both hash to 2236

How Java resolves it — chaining, then a tree

STEP 01Linked list
Colliding entries are chained in the same bucket (a singly linked list of Nodes). Lookup finds the bucket by hash, then walks the chain calling equals() to pick the right key.
↓ chain grows past 8 (and table ≥ 64)
STEP 02Red-black tree
Since Java 8, a bucket with > TREEIFY_THRESHOLD (8) nodes treeifies into a balanced red-black tree — lookups in that bucket go from O(n) to O(log n). It reverts to a list below UNTREEIFY_THRESHOLD (6).

Treeification only happens once the whole table is at least 64 entries; below that, Java resizes the table instead.

Performance & the contract that prevents pile-ups

  • Average lookup O(1); worst case O(n) with a list, O(log n) once treeified. A terrible hashCode() that returns a constant degrades the map to a linked list.
  • Load factor 0.75 (default): when size > capacity × 0.75 the table doubles and rehashes, spreading entries out and lowering collision density.
  • equals/hashCode contract: equal objects must have equal hash codes; override both together. Otherwise you store a key you can never retrieve.
i
Other resolution strategies (general CS): Java uses separate chaining. The alternative is open addressing — on collision, probe for another empty slot: linear probing (next slot), quadratic probing, or double hashing. (ThreadLocalMap uses linear probing.)
🎤 Say thisA collision is two keys landing in the same bucket — same hash, or different hashes that reduce to the same index. Java handles it by chaining the entries in a linked list and using equals() to tell them apart; since Java 8, a bucket that grows past 8 entries converts to a red-black tree, so the worst case is O(log n) instead of O(n). A good hashCode() and the 0.75 load-factor resize keep collisions rare.
What changed in Java 8?
Buckets that get too long (> 8 nodes, table ≥ 64) convert from a linked list to a red-black tree, capping worst-case bucket lookup at O(log n). Pre-8 it was always a list → O(n).
What if hashCode() always returns the same value?
Every key collides into one bucket. The map still works (equals disambiguates) but degrades to O(n) — effectively a linked list. It's a classic "why is my map slow" answer.
Why must you override equals() and hashCode() together?
Lookup first finds the bucket by hashCode(), then matches the key by equals(). If two "equal" objects have different hashes they land in different buckets and you'll never find the one you stored.
Chaining vs open addressing?
Chaining stores collisions in a per-bucket list/tree (Java's HashMap). Open addressing keeps everything in the array and probes for the next free slot (linear/quadratic/double hashing) — better cache locality, but sensitive to load factor and clustering.

Data · write one query

Write one SQL query

A single query that exercises the clauses interviewers actually probe: a JOIN, an aggregate, group-level filtering with HAVING, and ordering. The prompt: "For each department, show its headcount and average salary — but only departments with at least two employees, highest average first."

The schema

employees · departments

The query

JOIN · GROUP BY · HAVING · ORDER BY

Why each clause is there

ClauseDoes
LEFT JOINKeeps departments with zero employees too (they'd be dropped by an INNER JOIN). Filtered out later here by HAVING, but it makes the headcount honest.
COUNT(e.id)Counts non-NULL employee rows. Not COUNT(*) — with a LEFT JOIN an empty dept yields one all-NULL row, which COUNT(*) miscounts as 1.
GROUP BYCollapses rows into one per department so the aggregates compute per group.
HAVINGFilters groups after aggregation (COUNT(e.id) >= 2). WHERE can't — it runs before grouping.
ORDER BYSorts the final rows; it runs last, so it can use the avg_salary alias.
→
The clause-order insight they're testing: SQL is written SELECT…FROM…WHERE…GROUP BY…HAVING…ORDER BY but evaluated FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY. That's why WHERE can't see an aggregate (it runs before grouping) and ORDER BY can use a SELECT alias (it runs after).
🎤 Say thisI LEFT JOIN departments to employees so empty departments still appear, group by department, aggregate with COUNT and AVG, filter the groups with HAVING — not WHERE, because the filter is on an aggregate — and order by the average descending. I use COUNT(e.id), not COUNT(*), so a department with no employees counts as zero.
WHERE vs HAVING?
WHERE filters individual rows before grouping and can't use aggregates. HAVING filters groups after aggregation and can. You can use both in one query.
Why COUNT(e.id) instead of COUNT(*)?
COUNT(*) counts rows including the all-NULL row a LEFT JOIN produces for an empty department (→ 1). COUNT(e.id) ignores NULLs and correctly reports 0.
Make it "second-highest average" instead.
Wrap it in a CTE and rank: DENSE_RANK() OVER (ORDER BY avg_salary DESC), then filter WHERE rnk = 2 in the outer query.

Spring internals

What is the lifecycle of a bean?

A bean is an object the Spring IoC container creates, configures, and manages. Its lifecycle is the ordered set of steps from instantiation to destruction, with hooks you can plug into at the start (init) and end (destroy).

The phases, in order

PHASE 01Instantiate
Constructor runs. The container creates the raw object (this is where constructor injection supplies dependencies).
↓
PHASE 02Populate
Dependency injection of remaining properties — setter/field @Autowired values are set.
↓
PHASE 03Aware callbacks
If implemented: setBeanName(), setBeanFactory(), setApplicationContext() — the bean learns about its container.
↓
PHASE 04BPP before-init
BeanPostProcessor.postProcessBeforeInitialization() runs for every bean — the hook framework features use.
↓
PHASE 05Initialize
In order: @PostConstruct → InitializingBean.afterPropertiesSet() → custom initMethod. Do startup work once dependencies are wired.
↓
PHASE 06BPP after-init
postProcessAfterInitialization() runs — where Spring wraps the bean in a proxy for AOP / @Transactional.
↓ bean is ready & in use ↓
PHASE 07Destroy
On context shutdown: @PreDestroy → DisposableBean.destroy() → custom destroyMethod. Release resources here.

The three ways to hook init & destroy

annotations, interfaces, and @Bean methods — pick one style

Recommended order of preference: @PostConstruct/@PreDestroy (JSR-250, framework-neutral) > @Bean(initMethod, destroyMethod) (no Spring import in your class) > the InitializingBean/DisposableBean interfaces (couples you to Spring).

!
Prototype gotcha: Spring does not call destroy callbacks on @Scope("prototype") beans — it hands the bean over and forgets it. You're responsible for cleanup. Singletons get the full lifecycle.
🎤 Say thisThe container instantiates the bean (constructor injection here), injects the rest of its dependencies, calls any Aware interfaces, then runs BeanPostProcessors around the init phase — @PostConstruct, then afterPropertiesSet(), then a custom init method. The post-processor's after-init step is where Spring wraps it in a proxy for things like @Transactional. On shutdown it runs @PreDestroy, destroy(), then the custom destroy method — though prototype beans don't get destroy callbacks.
@PostConstruct vs afterPropertiesSet vs init-method — order?
All three run, in that order: @PostConstruct first, then InitializingBean.afterPropertiesSet(), then the custom initMethod. Same trio mirrored for destroy.
Where do AOP proxies / @Transactional get applied?
In BeanPostProcessor.postProcessAfterInitialization() — Spring returns a proxy that wraps your bean. That's also why self-invocation bypasses @Transactional: the call doesn't go through the proxy.
Are prototype beans fully managed?
No. Spring creates and wires them and runs init callbacks, but never calls destroy callbacks — the container doesn't keep a reference. Clean them up yourself.
What's a BeanPostProcessor good for?
It's the extension point that runs for every bean around initialization — used internally for @Autowired resolution, AOP proxying, and you can write your own to modify/wrap beans.

Spring core · the "D" in SOLID, productised

What is the purpose of IoC in Spring?

Inversion of Control is a principle: an object no longer creates or looks up its own dependencies — that control is inverted to a container. In Spring, the IoC container (the ApplicationContext) instantiates, configures, wires, and manages your objects ("beans"). Dependency Injection is how it does it.

Without IoC vs with IoC

the class controls creation — tight coupling
the container injects the dependency — loose coupling

The purpose — what it actually buys you

BenefitBecause…
Loose couplingCode depends on an interface, not a concrete class — swap implementations without editing callers (hello, Open/Closed).
TestabilityInject a mock/stub in a unit test; no Spring context needed with constructor injection.
Lifecycle mgmtThe container owns creation, singleton caching, and the init/destroy callbacks from tab 6.
Central configWiring lives in one place (component scan + @Configuration), not scattered new calls.
Cross-cuttingBecause beans are proxied, you get @Transactional, security, caching "for free."

How Spring injects — three flavours

  • Constructor injection (preferred) — dependencies are final, the object is always valid, trivially unit-testable. A single constructor needs no @Autowired.
  • Setter injection — for optional / reconfigurable dependencies.
  • Field injection (@Autowired on a field) — concise but can't be final and hides dependencies; avoid in new code.
→
IoC vs DI: IoC is the principle (the framework controls flow — "don't call us, we'll call you," the Hollywood Principle). DI is the concrete technique Spring uses to implement it. The container is Dependency Inversion (SOLID's "D") delivered as a feature.
🎤 Say thisIoC means I don't new my dependencies — the Spring container creates and injects them, usually through the constructor. The purpose is loose coupling and testability: my class depends on an interface and the container decides the implementation, so I can swap it or inject a mock without touching the class. It also lets Spring manage bean lifecycle and apply cross-cutting concerns like transactions via proxies. DI is just the mechanism that implements IoC.
IoC vs Dependency Injection — are they the same?
No. IoC is the broad principle (control of object creation/flow is handed to the framework). DI is one way to achieve it — supplying dependencies from outside. Spring's container does IoC via DI.
Why prefer constructor injection?
Fields can be final (immutable, thread-safe), the object can't exist half-wired, required dependencies are explicit, and you can instantiate it in a plain unit test with new + mocks — no container.
BeanFactory vs ApplicationContext?
Both are IoC containers. BeanFactory is the basic lazy one; ApplicationContext extends it with eager singleton instantiation, event publishing, i18n, and annotation/AOP support — it's what you actually use.
How does the container know what to inject?
Component scanning finds @Component/@Service/@Repository classes (and @Bean methods), registers them as bean definitions, then autowires by type (falling back to name; disambiguate with @Qualifier/@Primary).

Memory hooks

  • SOLID — the word is the mnemonic. Order it as a sentence: "Some One Likes Inverting Dependencies."
  • Open/Closed — picture a power strip: you extend by plugging in new devices, you never rewire the strip.

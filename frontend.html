Signatures, minimal correct examples, and the gotcha an interviewer probes for. React 19 stable. Redux via Redux Toolkit (the official default — plain createStore is legacy). Press / to filter, ← → to move between sections.

01

Core hooks

react

useState

Local reactive state. Setter triggers re-render.

const [count, setCount] = useState(0);
// functional update — use when next depends on prev
setCount(c => c + 1);
// lazy init — fn runs once, not every render
const [v] = useState(() => expensiveInit());
!Gotcha: state updates are async & batched. Reading count right after setCount gives the old value. Multiple setCount(c=>c+1) in one event stack correctly; setCount(count+1) twice does not.

useEffect

Synchronize with external systems after render (subscriptions, fetch, DOM).

useEffect(() => {
  const id = setInterval(tick, 1000);
  return () => clearInterval(id); // cleanup
}, [dep]); // deps control re-run
!Deps: [] = once on mount; omitted = every render; [a,b] = when a or b change (Object.is). Cleanup runs before next effect & on unmount. In StrictMode dev, effects run twice — your cleanup must make that safe.

useContext

Read a context value without prop drilling.

const ThemeCtx = createContext('light');
// provider
<ThemeCtx.Provider value={theme}>…</ThemeCtx.Provider>
// consumer
const theme = useContext(ThemeCtx);
!Gotcha: every consumer re-renders when the provider value changes by identity. Passing {a,b} inline creates a new object each render → memoize the value. Not a substitute for a real state manager when updates are frequent.

useReducer

State via reducer — for complex/related state or when next state depends on prior.

function reducer(state, action){
  switch(action.type){
    case 'inc': return {n: state.n + 1};
    default: return state;
  }
}
const [state, dispatch] = useReducer(reducer, {n:0});
dispatch({type:'inc'});
?vs useState: reach for it when state transitions are non-trivial, multiple sub-values move together, or you want logic testable in isolation. Same mental model as Redux, scoped to one component.

useRef

Mutable box that persists across renders without causing re-renders.

const inputRef = useRef(null);
inputRef.current.focus();        // DOM access
const renders = useRef(0);
renders.current++;               // instance variable
!Key idea: mutating .current does not trigger a render. Use for DOM nodes, timer IDs, previous values, any "I need to remember this but it shouldn't paint." Don't read/write .current during render.
02

Performance hooks

react

useMemo

Cache a computed value between renders.

const sorted = useMemo(
  () => bigList.sort(cmp),
  [bigList]            // recompute only when deps change
);
!Use when: the computation is genuinely expensive, or the result is a referential dependency for another hook / React.memo child. Don't wrap every value — memoization has its own cost. It's an optimization, not a semantic guarantee.

useCallback

Cache a function identity between renders. useCallback(fn,d) ≡ useMemo(()=>fn,d).

const onAdd = useCallback(
  (item) => setList(l => [...l, item]),
  []                 // stable identity
);
<MemoChild onAdd={onAdd} />
!Only matters when the function is passed to a memoized child or used in a dependency array. Wrapping a handler that goes to a plain DOM element buys nothing.

React.memo

HOC that skips re-render if props are shallow-equal. (Not a hook, but its partner.)

const Row = React.memo(function Row({label}){…});
// custom comparator (rare)
React.memo(Row, (prev,next) => prev.id===next.id);
?The trio: memo stops the child re-rendering; useCallback/useMemo keep its props referentially stable so memo actually hits. One without the others is usually pointless.

React Compiler (context)

Worth a sentence in interviews.

  • The React Compiler (RC, 2024→) auto-memoizes components, reducing the need for manual useMemo/useCallback/memo.
  • Correct answer to "how do you optimize re-renders?" now includes: measure first (Profiler), fix structure (lift/colocate state, split components, pass children), then memoize.
03

Advanced hooks

react

useLayoutEffect

Like useEffect but fires synchronously after DOM mutation, before paint.

useLayoutEffect(() => {
  const {height} = ref.current.getBoundingClientRect();
  setH(height);              // measure before user sees it
}, []);
?vs useEffect: use only when you must read layout and re-render before paint to avoid a visible flicker. Blocks paint, so it's slower — default to useEffect.

useImperativeHandle

Customize the ref instance a parent receives.

const Input = forwardRef((props, ref) => {
  const inner = useRef();
  useImperativeHandle(ref, () => ({
    focus: () => inner.current.focus()
  }));
  return <input ref={inner} />;
});
!Escape hatch — exposes imperative methods. In React 19 ref is a normal prop, so forwardRef is no longer required.

useId

Stable unique id, SSR-safe (server & client match).

const id = useId();
<label htmlFor={id}>Name</label>
<input id={id} />
!For accessibility wiring, not for list keys. Never use it as a React key.

useDebugValue

Label custom hooks in React DevTools.

useDebugValue(isOnline ? 'Online' : 'Offline');
iLow-frequency interview topic — recognize it, move on.
04

Concurrent hooks

react 18+

useTransition

Mark a state update as non-urgent so it doesn't block input.

const [isPending, startTransition] = useTransition();
startTransition(() => {
  setQuery(input);   // low priority, interruptible
});
{isPending && <Spinner/>}
?Why: keeps the typed input responsive while an expensive list filters. Urgent update (the keystroke) wins; the transition is deferred/interrupted.

useDeferredValue

Render a "lagging" copy of a value at lower priority.

const deferred = useDeferredValue(query);
const list = useMemo(
  () => filter(items, deferred), [deferred]
);
?vs useTransition: useTransition wraps the update you trigger; useDeferredValue defers a value you receive (e.g. a prop you don't control). Same goal, different access point.

useSyncExternalStore

Subscribe to an external (non-React) store safely under concurrency.

const state = useSyncExternalStore(
  subscribe,        // (cb) => unsubscribe
  getSnapshot,      // () => current value
  getServerSnapshot // optional, for SSR
);
iThe hook library authors use (react-redux, Zustand) to avoid tearing. You rarely call it directly, but know why it exists: consistent reads during concurrent renders.
05

React 19 hooks

react 19

use

Read a Promise (suspends) or Context. May be called conditionally.

function Comments({promise}){
  const data = use(promise);   // suspends until resolved
  return <List items={data} />;
}
// also: const theme = use(ThemeCtx);
!The one "hook" exempt from the call-at-top rule — use is allowed inside conditionals/loops. Suspends, so pair with <Suspense> and an error boundary.

useActionState

State + pending for a form Action (replaces useFormState).

const [state, formAction, isPending] =
  useActionState(async (prev, formData) => {
    return await submit(formData);
  }, initialState);
<form action={formAction}>…</form>
iPart of React "Actions" — async form submission with built-in pending/error/optimistic flow, less manual useState plumbing.

useFormStatus

Read the parent <form>'s submission status. From react-dom.

const {pending, data, method} = useFormStatus();
return <button disabled={pending}>
  {pending ? 'Saving…' : 'Save'}
</button>;
!Must be in a component rendered inside the form — it reads context, not its own props. Common gotcha: calling it in the same component that renders the <form> returns nothing.

useOptimistic

Show an optimistic UI state while an async action is in flight.

const [optimistic, addOptimistic] =
  useOptimistic(messages, (state, newMsg) =>
    [...state, {...newMsg, sending:true}]);
addOptimistic(draft);   // UI updates instantly
await sendMessage(draft); // reverts/reconciles on settle
iReverts automatically to the real state once the action resolves. Pairs with useActionState / form Actions.

Rules of hooks & custom hooks

react

Rules of Hooks

  • Only call at the top level — never inside conditions, loops, or nested functions. (Exception: use.)
  • Only call from React function components or other custom hooks.
  • Why: React tracks hook state by call order, not name. Conditional calls shift the order and corrupt the mapping.
!Classic interview trap: "why can't you put useState in an if?" → because hook identity is positional across renders.

Custom hooks

Extract reusable stateful logic. Name must start with use.

function useToggle(init=false){
  const [on, setOn] = useState(init);
  const toggle = useCallback(() => setOn(o => !o), []);
  return [on, toggle];
}
const [open, toggleOpen] = useToggle();
?Sharing a hook shares logic, not state — each call gets its own isolated state. To share state, lift it to context or a store.
07

React essentials (non-hook, frequently asked)

react

Keys & reconciliation

  • Keys must be stable, unique among siblings. Index-as-key breaks on reorder/insert/delete (wrong element reused, lost input state).
  • React diffs the virtual DOM by type + key to decide reuse vs remount.
  • Changing a component's key forces a full remount — a legit reset trick.

Controlled vs uncontrolled

// controlled — React owns the value
<input value={v} onChange={e=>setV(e.target.value)} />
// uncontrolled — DOM owns it, read via ref
<input defaultValue="x" ref={ref} />
?Controlled = single source of truth, validation, conditional UI. Uncontrolled = simpler, less re-render, file inputs.

Lazy loading & Suspense

const Chart = lazy(() => import('./Chart'));
<Suspense fallback={<Spinner/>}>
  <Chart />
</Suspense>
?Code-splitting at route/component boundaries cuts initial bundle. Suspense also coordinates data fetching (use) and shows fallbacks.

Error boundaries

Catch render errors in a subtree. Still class-only.

class Boundary extends React.Component{
  state={err:null};
  static getDerivedStateFromError(e){return {err:e}}
  componentDidCatch(e,info){log(e,info)}
  render(){return this.state.err ? <Fallback/> : this.props.children}
}
!No hook equivalent yet. They do not catch errors in event handlers, async code, or SSR — only in render/lifecycle of children.

Redux — core concepts

redux

The one-way data flow

UI dispatch(action) ─▶ reducer(state, action) ─▶ new state ─▶ store ─▶ UI
                                  ▲ pure, no mutation, no side effects
TermWhat it is
storeSingle immutable state tree + dispatch.
actionPlain object {type, payload} describing what happened.
reducerPure (state, action) => newState. No mutation, no I/O, no randomness.
dispatchSends an action to the store.
selectorstate => derivedValue. Reads/computes from the tree.
middlewareIntercepts dispatch (logging, async thunks).
!Three principles: single source of truth · state is read-only (change only via actions) · changes via pure reducers.
09

React-Redux hooks

react-redux

useSelector

Read & subscribe to a slice of store state.

const count = useSelector(state => state.counter.value);
// re-renders when the selected value changes (===)
!Gotcha: returning a new object/array each call (state => ({a,b})) breaks the default === equality → re-render every dispatch. Fix with shallowEqual as the 2nd arg, or a memoized selector (createSelector), or select primitives separately.

useDispatch

Get the store's dispatch function.

const dispatch = useDispatch();
dispatch(increment());          // action creator
dispatch(addTodo({text:'x'}));
idispatch identity is stable — safe to omit from / include in dep arrays without re-running.

useStore

Access the store instance directly. Rarely needed.

const store = useStore();
store.getState();
iEscape hatch for reading state in a callback without subscribing. Prefer useSelector; reach here only when you explicitly do not want a subscription.

Provider setup

import {Provider} from 'react-redux';
<Provider store={store}>
  <App />
</Provider>
?Under the hood react-redux uses useSyncExternalStore to subscribe each connected component to the store safely under concurrent rendering.

Redux Toolkit (RTK)

official default

configureStore

Sets up store + good defaults (thunk, DevTools, immutability checks).

const store = configureStore({
  reducer: { counter: counterReducer, todos: todosReducer }
});
iReplaces createStore + manual middleware/devtools wiring. Includes redux-thunk by default.

createSlice

Reducer + action creators + types in one shot.

const counter = createSlice({
  name: 'counter',
  initialState: {value: 0},
  reducers: {
    increment: (s) => { s.value++ },        // "mutate" — Immer
    addBy: (s, action) => { s.value += action.payload }
  }
});
export const {increment, addBy} = counter.actions;
export default counter.reducer;
!The "mutation" question: RTK uses Immer. You write mutating syntax against a draft; Immer produces a new immutable state. You may mutate the draft or return a new value — never both.

createAsyncThunk

Async logic → auto pending/fulfilled/rejected actions.

const fetchUser = createAsyncThunk(
  'users/fetch',
  async (id) => (await api.get(id)).data
);
// in slice:
extraReducers: (builder) => {
  builder
    .addCase(fetchUser.pending,   s => { s.loading = true })
    .addCase(fetchUser.fulfilled, (s,a) => { s.loading=false; s.data=a.payload })
    .addCase(fetchUser.rejected,  (s,a) => { s.loading=false; s.error=a.error });
}
?Why thunks: reducers must be pure, so side effects (fetch) live in thunks/middleware, which dispatch plain actions on completion.

createSelector (reselect)

Memoized derived data — recomputes only when inputs change.

const selectDone = createSelector(
  [state => state.todos],
  (todos) => todos.filter(t => t.done)  // cached
);
!Prevents the "useSelector returns a new array every render" re-render problem for computed/filtered data.

RTK Query

data fetching

createApi

Declarative data layer — caching, dedup, invalidation, auto-generated hooks.

const api = createApi({
  reducerPath: 'api',
  baseQuery: fetchBaseQuery({baseUrl:'/api'}),
  tagTypes: ['Post'],
  endpoints: (b) => ({
    getPosts: b.query({ query: () => 'posts', providesTags:['Post'] }),
    addPost:  b.mutation({ query: body => ({url:'posts',method:'POST',body}),
                           invalidatesTags:['Post'] })
  })
});
export const {useGetPostsQuery, useAddPostMutation} = api;

Generated hooks

// query
const {data, error, isLoading, isFetching, refetch} =
  useGetPostsQuery();
// mutation
const [addPost, {isLoading: saving}] = useAddPostMutation();
await addPost({title:'Hi'}).unwrap();   // throw on error
?When to mention: if asked "how do you handle server state in Redux," RTK Query is the modern answer — it replaces hand-written thunks + loading-flag reducers for API data, and overlaps with React Query / SWR conceptually.

State management — decision guide

judgment
NeedReach for
State used by one component / a couple of childrenuseState / lift state up
Complex transitions in one componentuseReducer
Low-frequency global value (theme, locale, auth user)Context + useState/useReducer
Large app, frequent global updates, time-travel/debug, many consumersRedux Toolkit
Server/API data (caching, refetch, invalidation)RTK Query / React Query — not hand-rolled Redux
Lightweight global store, less boilerplateZustand / Jotai (know they exist)
!The mature answer: "I don't default to Redux. Most state is local. I separate server state (RTK Query/React Query) from client/UI state (local, or Redux when it's genuinely global and shared). Context is for low-churn values, not a state manager." Saying "I always use Redux" is a red flag to senior interviewers.

Gotchas & rapid-fire Q&A

drill

Stale closure

The #1 hooks bug.

useEffect(() => {
  const id = setInterval(() => {
    setCount(count + 1);   // ❌ count frozen at mount value
  }, 1000);
  return () => clearInterval(id);
}, []);                    // empty deps → closure captures count=0
!Fix: functional updater setCount(c => c + 1), or add count to deps, or hold it in a ref. Be ready to explain why: the effect closes over the render's variables.

Batching

setA(1); setB(2);   // one re-render (React 18 batches everywhere,
                    // incl. promises, timeouts, native events)
iPre-18, updates outside React event handlers weren't batched. React 18 automatic batching changed that.

Why do hooks have to run in the same order every render?

React stores hook state in a list keyed by call index, not name. Conditional calls would desync that list with state.

useMemo vs useCallback?

Same machine. useMemo caches the return value; useCallback caches the function itself. useCallback(fn,d) === useMemo(()=>fn,d).

useEffect vs useLayoutEffect?

useEffect runs after paint (async, non-blocking). useLayoutEffect runs after DOM mutation but before paint (sync) — use to measure/mutate layout without flicker.

Why must reducers be pure?

Predictability, time-travel debugging, and React may call them multiple times. Side effects go in thunks/middleware, not reducers.

How does RTK let you "mutate" state?

Immer wraps the draft; mutations are recorded and applied to a fresh immutable copy. The actual store state is never mutated.

Context vs Redux?

Context is a dependency-injection mechanism (passes a value down), not a state manager. It re-renders all consumers on change and has no dev tooling, middleware, or selector memoization. Redux adds those for frequently-updated shared state.

Why is index a bad key?

On reorder/insert/delete, indices shift, so React maps the wrong DOM nodes — corrupted state, wrong animations, lost focus. Use a stable id.

How do you stop unnecessary re-renders?

Measure with Profiler first. Then: colocate/lift state correctly, split components, pass children, memoize selectors, then React.memo/useMemo/useCallback — or rely on the React Compiler.

Can you call hooks in a custom hook?

Yes — custom hooks are the one place besides components where hooks may be called. They share logic, not state.
